# U3 · STEP 01 ｜ 訂單看板：讓資料「活」起來

> **這堂完成物**：看懂一間「營業中」的店怎麼把資料變成畫面，再到 LINE 推播中心走完 Flex mock 流程。

U2 練的是控制 AI 能寫什麼 code。今天換一件事：**控制行動**——human-in-the-loop 閘門 + token 工具邊界，決定 AI 能不能對真實世界採取行動（送出一則真的 LINE 通知）。下一堂 U4 是再擴大一步：讓 AI 能碰到更多外部工具。

## 1. 開始營業，看訂單「活」起來

啟動專案後，點上方「**訂單看板**」，按「**開始營業**」。

你應該看到：

- 訂單持續進來、狀態自己往前走（待製作 → 製作中 → 待取餐 → 已取餐）
- 上方 KPI（今日訂單數、進行中、缺料等待、已收營業額）跟著跳動
- 下方「備料水位」的庫存數字自動往下掉

「暫停」讓畫面凍結方便你細看；「重置」讓你隨時回到同一個乾淨起點。

## 2. 打開 DevTools，看資料怎麼來的

按 F12 → Network，看畫面每 3 秒發出的請求：

```text
GET /api/orders
```

這就是 API：前端主動去問後端「有新資料嗎」。回應內容是 JSON——訂單清單、庫存、推播建議都在裡面，這就是 payload。

這個 API **只活在 `npm run dev`**：如果你在 `npm run build/preview` 或後端沒開，看板會自動退回 `shopData.js` 的靜態範例，不會白屏，但也不會動。

## 3. 自己「改個數字」，看警示怎麼出現

在「備料水位」找一項原料，按「**盤點 −5**」，故意把它改到低於安全量。

你應該看到：

- 那一列從「正常」變成「需要補貨」（或改到負數會變「數字有誤」）
- 上方的推播建議跟著多一條「庫存低於安全量 → 推播中心『庫存警示』」

這就是「後台資料一直有進來、系統自動判斷」的自動化管理平台：資料會變，判斷規則不變。

## 4. 認識三個前端名詞

| 名詞 | 在這個專案裡看哪裡 |
|---|---|
| component | `web-lab/src/OrderBoard.jsx` 裡的 `OrderLane` |
| CSS | `web-lab/src/styles.css` 裡的 `.order-lane`、`.kpi-card`、`.inv-row` |
| src | `web-lab/src/`，前端主要程式都在這 |

學生不需要從零寫 React，但要看得懂：資料變了，component 和 CSS 會讓畫面跟著變。

## 5. 切到「LINE 推播中心」

點「**LINE 推播中心**」，最上面有三種訊息類型：

- 訂單資訊（藍）——優先讀看板的即時訂單，沒開店退回 `data-lab/orders.json`
- 庫存警示（琥珀）——伺服端依「現在的庫存」即時組成
- 營運異常（紅）——讀 `data-lab/report.json`

三種都走同一套流程：

```text
載入資料 → 檢查資料合約 → Flex 視覺預覽 → 人工審核 → mock 推播
```
